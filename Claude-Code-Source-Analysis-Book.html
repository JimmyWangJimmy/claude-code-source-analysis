7 大核心 Prompt 策略

#策略洞察
1先读后改强制模型修改前先获取当前状态，解决"幻觉式编码"
2反过度工程"三行重复代码好过一个过早抽象"
3安全确认一次授权 ≠ 永久授权，防止 Agent 越权
4不给时间估计LLM 无法准确估时，不说比说错好
5安全编码OWASP Top 10 意识提醒
6不暴力重试禁止无脑重试，要求分析原因后换方法
7不兼容 Hack不留 _unused、// removed 等残留

Prompt 优先级

Override (最高) → Coordinator → Agent → Custom → Default → Append (最低)

最重要的 5 个洞察

1. Prompt 是产品 — 投入与代码同等精力
2. 安全是多层的 — Deny→Allow→Hook→Classifier→User
3. 上下文是有限的 — 工具结果管理+压缩+存盘=一整套工程
4. 子 Agent 需要隔离 — 权限、工具、上下文逐级递减
5. 性能源于架构 — 预取、懒加载、缓存分段、cache 共享

14 条设计启示

#启示核心要点
1多 Agent 是架构决策从第一天设计进去，不是后加功能
2Agent 是资源持有者必须有 try/finally 清理机制

4AbortController 链取消信号单向向下传播
5四种 Agent 模式Subagent/Fork/Teammate/Coordinator 各有场景
6Fork Cache 共享字节对齐请求前缀，节省 ~75% input token
7分层通信小数据→消息、大数据→文件、协调→结构化消息
8权限逐级递减子 Agent 不继承父的权限批准
9声明式 Agent 定义Markdown frontmatter 定义 Agent 类型
10Scratchpad 模式共享目录交换大数据
11防递归三层防护Prompt + 代码标记 + 深度限制
12输出格式约束子 Agent 输出给程序消费，必须结构化
13资源预算约束Token/轮次/时间/边际收益缺一不可
14三级记忆架构Agent 类型 × 作用域 二维隔离

隔离矩阵 (最精妙的设计)

资源策略原因
AbortController独立，父 abort 向下传播取消父→取消子，取消子≠取消父
文件状态缓存深拷贝防止并发写冲突
权限决策全新子不继承父的批准
工具列表可配置子集最小权限原则
消息历史Fork 继承 / Subagent 独立按需决定
记忆按 Agent 类型隔离不同类型积累不同经验
